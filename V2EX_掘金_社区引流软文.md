# 【OpenClaw + Mem0 终极方案】彻底解决 AI 新会话失忆，构建长期记忆闭环

我受够了一件事：每次新建对话，AI 就把我忘了。

前一轮刚告诉它“我的公司名字、我的偏好、我的任务规则”，下一轮 `/new` 之后就像第一次见面。对做自动化助手的人来说，这种“会话级失忆”基本不可接受。

这次我干脆不在前端打补丁了，直接从后端下手，把 OpenClaw 改成了“Mem0 第一记忆源”模式，目标很明确：

1. 用户输入一进来，立刻入库（语义级）
2. 每次发给大模型前，必须先检索历史记忆
3. 回复结束后，把“用户问题 + AI 回答”再回写
4. `/new` 只清上下文，不清长期记忆

最终把整条链路跑通了，稳定可用。

---

## 这套方案解决了什么

传统 OpenClaw 常见痛点：

- 依赖 `soul.md` / `USER.md` / 当前上下文，长对话后容易漂
- 新会话后记忆连续性差
- 多终端（Mac/Linux/飞书）记忆割裂

这套改造后的效果：

- 强制后端记忆检索，不再“看心情记住”
- Mem0 向量存储 + 语义召回
- 多环境可共享同一记忆（统一 `user_id` 即可）
- 记忆链路与模型链路解耦，可从稳态配置平滑升级到高性能部署

---

## 硬核思路（核心干货）

我做的不是 prompt 工程，而是服务端链路重构：

1. 在 OpenClaw 插件层拦截 `message_received`，异步写 Mem0。
2. 在 `before_prompt_build` 里强制 `memory/search`，把结果注入系统消息顶部：`[历史记忆：...]`。
3. 在 `agent_end` 回写“问答对”。
4. 屏蔽旧 memory 工具调用（避免 `node:sqlite` 那套老链路干扰）。
5. 修复 `/new` 后 user_id 漂移问题，保证同一用户始终查同一记忆空间。

技术栈：

- OpenClaw（网关）
- Mem0 + FastAPI（本地 API）
- `BAAI/bge-large-zh-v1.5`（中文向量模型）
- Kimi K2.5（LLM）
- systemd（服务托管）

高性能扩展也留好了接口：可把 embedding 服务独立出去，并升级到 `BAAI/bge-m3`，在多节点下提升检索质量与吞吐。

## 多端共享记忆到底能不能实现

能。

只要满足三点：

1. 多个 OpenClaw 实例指向同一个 Mem0 后端
2. 所有节点 embedding 模型一致
3. `user_id` 统一映射（不要直接用渠道临时 ID）

实战里推荐用 sidecar：每台机器本地 `127.0.0.1` 收请求，再转发到中心 Mem0。这样本地低延迟和中心共享都能兼顾。
